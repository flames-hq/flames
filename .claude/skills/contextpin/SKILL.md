# ContextPin CLI Skill

Use the `ctxpin` CLI to create, inspect, and manage structured notes in a ContextPin workspace.

## Critical Rules

1. **Never create note files manually.** Always use `ctxpin notes create note` — it is the only way to get a valid ID (10-char nanoid). Manual creation risks missing IDs, duplicates, or malformed frontmatter.
2. **Always use labels from `.contextpin/settings.json`.** Labels support wildcard patterns (`SPEC-*`, `*-DONE`). Check the workspace settings before assigning labels.
3. **Edit note content after creation.** The CLI creates the file with correct frontmatter; then edit the markdown body to add content. Add labels to the frontmatter `labels:` array.

## Commands Reference

### Note & Folder Operations

```bash
# Create a note (generates valid ID, frontmatter, timestamps)
ctxpin notes create note "<Name>" [--folder "<Folder>"] [--json]

# Create a folder
ctxpin notes create folder "<Name>" [--parent "<Parent>"] [--json]

# List notes and folders (defaults to root)
ctxpin notes list [PATH] [--json]

# Show full info for a note or folder by ID
ctxpin notes show <ID> [--json]
```

### Utility Commands

```bash
# Resolve any ID or contextpin:// URI to its file path
ctxpin resolve <ID> [--json]

# Validate notes, detect issues (missing/duplicate IDs, broken refs)
ctxpin doctor [--json]

# Query the running ContextPin desktop app for active context
ctxpin ade show [--format claude|codex] [--json]
```

### Global Options

```bash
--workspace <PATH>   # Override workspace root (skip auto-discovery)
-h, --help           # Print help
```

## Workflow

### Creating Notes

```bash
# 1. Create the note via CLI
ctxpin notes create note "Architecture Overview" --folder "Specs" --json

# 2. The CLI outputs the file path and contextpin:// URI
# 3. Edit the created file to add content and labels
```

### Note File Format

Notes are markdown files with YAML frontmatter (auto-generated by CLI):

```markdown
---
schema_version: 1
id: d60VNl4JrI
created_at: 2026-03-27T22:19:13Z
updated_at: 2026-03-28T15:49:17Z
labels:
- SPEC-ARCH
- DOCS
position: 1.0
---

# Note Title

Content here...
```

### Frontmatter Fields

| Field            | Type     | Required | Description                        |
|------------------|----------|----------|------------------------------------|
| `schema_version` | integer  | Yes      | Always `1`                         |
| `id`             | string   | Yes      | 10-char alphanumeric nanoid        |
| `created_at`     | ISO 8601 | Yes      | Creation timestamp (UTC)           |
| `updated_at`     | ISO 8601 | Yes      | Last modification timestamp (UTC)  |
| `labels`         | string[] | No       | Tags for categorization            |
| `position`       | float    | No       | Custom ordering within folder      |

### Label Rules (`.contextpin/settings.json`)

Labels are styled via `label_rules` with pattern matching:

- **Exact match:** `"BUG"` matches `"BUG"` or `"bug"` (case-insensitive)
- **Wildcard prefix:** `"SPEC-*"` matches `"SPEC-AUTH"`, `"SPEC-123"`, etc.
- **Wildcard suffix:** `"*-DONE"` matches `"UI-DONE"`, `"API-DONE"`, etc.

Always check the workspace's `label_rules` before assigning labels.

### Folders

Folders use a `_frontmatter.yml` file for metadata:

```yaml
id: kY6FObqmJp
position: 2.0
```

### contextpin:// URIs

Format: `contextpin://<10-char-id>`. These are rename-safe and move-safe — they reference the stable ID, not the file path. Use them in cross-references between notes, source code comments, and commit messages.

## Diagnostics

Run `ctxpin doctor` to detect:
- Missing IDs
- Duplicate IDs
- Broken `contextpin://` references
- Malformed frontmatter

## Integration

ContextPin hooks into AI coding agents via `UserPromptSubmit` hooks:

```json
{
  "hooks": {
    "UserPromptSubmit": [{
      "command": "ctxpin ade show --format claude"
    }]
  }
}
```
